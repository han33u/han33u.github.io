# CHANGE_REQUEST

이 문서는 배포된 GitHub Pages 웹사이트에 대해 사용자가 전달한 수정 요청을 `Change Item` 단위로 분해하고, 수정·검증·배포까지 추적하기 위한 기준 문서입니다.

이 저장소의 변경은 한 번에 큰 덩어리로 처리하지 않고, 다음 원칙으로 관리합니다.

- 사용자의 요청 원문을 최대한 보존한다.
- 한 요청을 하나 이상 `Change Item`으로 분해한다.
- 한 `Change Item`은 가능한 한 한 가지 중심 수정만 담는다.
- 한 번에 하나의 실패 원인만 고친다.
- 기존에 통과한 기능은 깨뜨리지 않는다.
- 변경 전후 상태는 AORR과 MEMORY에 남긴다.

## 1. 현재 기준 상태

변경 요청을 시작하기 전에 다음을 먼저 확인한다.

- 배포된 GitHub Pages 웹사이트
- 현재 테스트 코드와 파일 구조
- Git remote와 현재 브랜치
- `AORR.md`
- `MEMORY.md`
- 마지막 정상 배포 commit
- 마지막 정상 배포 URL
- 기존 테스트 실행 기록
- 사용자가 제공한 참고 파일
- 현재 Git 상태

현재 기준 상태는 마지막으로 정상 배포된 화면과 기능을 기준으로 잡는다.

## 2. 사용자 요청 보존 규칙

사용자의 수정 요청은 가능한 한 원문에 가깝게 보존한다.

다만 다음 상황에서는 요청을 재구성할 수 있다.

- 요청이 너무 길어서 실행 단위로 나눠야 하는 경우
- 한 번에 여러 기능이 섞여 있는 경우
- 모호한 표현이 있어 항목별로 분리해야 하는 경우

이때도 원래 요청의 의미는 유지해야 한다.

## 3. Change Item 분해 규칙

각 변경 요청은 하나 이상의 `Change Item`으로 나눈다.

예시 형식:

- CR-001
- CR-002
- CR-003

`Change Item`은 다음 기준으로 분해한다.

- 콘텐츠 수정
- 문구 수정
- 디자인 수정
- 레이아웃 수정
- 반응형 수정
- 접근성 수정
- 게임 수정
- 성능 수정
- 구조 수정
- 배포 관련 수정

한 `Change Item` 안에는 가능한 한 하나의 중심 변경만 넣는다.

## 4. Change Item 필드

각 `Change Item`은 아래 항목을 반드시 포함한다.

- Change Item ID
- 사용자 요청 원문
- 요청 요약
- 요청 분류
- 현재 상태
- 기본 상태
- 수행할 작업
- 근거 자료
- 수정 대상 파일
- 변경 허용 범위
- 변경 금지 범위
- 테스트 방법
- 테스트 범위
- 예상 위험
- 배포 요약 필요 여부
- 사람 확인 필요 항목

필요하면 아래 항목도 추가할 수 있다.

- 우선순위
- 예상 난이도
- 롤백 기준
- 후속 작업

## 5. 요청 분류

각 `Change Item`은 다음 분류 중 하나 이상으로 표시한다.

- BUG
- UI_UX
- RESPONSIVE
- PERFORMANCE
- CONTENT
- DOCUMENT_BASED_CONTENT
- INFORMATION_ARCHITECTURE
- MULTI_PAGE_STRUCTURE
- NAVIGATION
- GAME_LOGIC
- GAME_CONTROL
- GAME_ENTITY
- GAME_EFFECT
- GAME_STATE
- ACCESSIBILITY
- NEW_FEATURE
- SPEC_CHANGE
- REFACTOR
- DEPLOYMENT
- SECURITY
- UNKNOWN

필요하면 여러 분류를 함께 달 수 있다.

## 6. 문서 기반 콘텐츠 처리

사용자가 CV, 이미지, PDF, 문서, 프롬프트 파일 같은 자료를 보내면 다음 순서로 처리한다.

1. 문서가 실제로 존재하는지 확인한다.
2. 문서에서 확인 가능한 사실만 추출한다.
3. 이름, 역할, 경력, 프로젝트, 연락처를 구분한다.
4. 어떤 정보가 빠졌는지 확인한다.
5. 문서에 없는 사실은 만들지 않는다.
6. 불확실한 내용은 `[사람 확인 필요]`로 둔다.
7. 개인정보나 민감 정보는 HITL 조건으로 분류한다.

이 절차는 포트폴리오, 게임, 디자인, 콘텐츠 편집 모두에 동일하게 적용한다.

## 7. 현재 문서에서 자주 다루는 요청 유형

이 저장소에서는 다음과 같은 요청이 자주 들어올 수 있다.

- 사이트 디자인 변경
- 프로필 문구 변경
- 트레일 러닝 콘셉트 반영
- 프로젝트 카드 재작성
- 게임 섹션 수정
- 게임 조작 개선
- 모바일 레이아웃 개선
- GitHub Pages 반영
- 콘텐츠 프롬프트 파일 추가

각 요청은 별도 `Change Item`으로 분리해 관리한다.

## 8. Change Request Loop

이 저장소의 변경은 다음 루프로 진행한다.

1. 요청 수집
2. 요청 해석
3. Change Item 분해
4. 우선순위 결정
5. 최소 작업 선정
6. 파일 수정
7. 검증
8. 필요 시 재수정
9. AORR 기록
10. MEMORY 갱신
11. 배포 확인

각 루프는 다음 상태 중 하나로 표현한다.

- CHANGE_INTAKE
- CHANGE_PLANNED
- READY
- ACTING
- VERIFYING
- RETRYING
- PASSED
- BLOCKED
- HITL_REQUIRED
- DEPLOY_APPROVAL_REQUIRED
- DEPLOYED

## 9. AORR에 남길 내용

변경 루프가 있을 때마다 AORR에는 다음을 남긴다.

- Loop ID
- 연결된 Change Item
- Target
- 입력 자료
- Act
- Observe
- Reason
- Verifier
- 완료 기준
- Retry 횟수
- Stop 조건
- HITL 조건
- 예상 수정 파일
- 수행한 Loop
- 다음 Loop

## 10. MEMORY에 남길 내용

MEMORY는 전체 프로젝트의 현재 상태와 다음 행동을 기억하는 용도다.

변경 요청이 진행될 때마다 MEMORY에는 다음을 갱신한다.

- 마지막 정상 배포 commit
- 마지막 정상 배포 URL
- 새로운 Change Request ID
- Change Item 목록
- 사용자 요청 요약
- 참고 자료
- 현재 상태
- 다음 완료 기준
- 루프 실행 순서
- 롤백 기준
- 사람 확인 필요 항목

## 12. 요청 분해 예시

사용자가 다음과 같이 요청했다고 가정한다.

- 프로필 문구를 바꿔 달라
- 트레일 러닝 느낌을 더해 달라
- 게임 버튼을 더 알아보기 쉽게 해 달라
- 모바일에서 레이아웃이 덜 답답하게 보여 달라

이 경우 한 번에 처리하지 말고 다음처럼 나눈다.

- CR-001: 프로필 문구 수정
- CR-002: 트레일 러닝 콘셉트 반영
- CR-003: 게임 버튼 UI 개선
- CR-004: 모바일 레이아웃 개선

각 항목은 독립적으로 검토하고 수정한다.

## 13. 권장 실행 원칙

이 저장소의 변경 작업은 다음 원칙을 따른다.

- 한 번의 Retry에서는 하나의 원인만 수정한다.
- 관련된 최소 파일만 수정한다.
- 기존 테스트를 삭제하거나 완화하지 않는다.
- 대규모 재작성은 피한다.
- 이미 통과한 기능은 깨뜨리지 않는다.
- 불확실한 정보는 추측하지 않는다.
- 배포가 필요하면 배포 전 검증을 먼저 한다.

## 14. 산출물 관리

이 변경 체계에서 유지할 문서는 다음과 같다.

- `AORR.md`
- `MEMORY.md`
- `CHANGE_REQUEST.md`
- 필요 시 `CONTENT_PROMPT.md`
- 필요 시 `TRAIL_RUNNING_CONTENT_PROMPT.md`

이 문서들은 서로 연결되어 있어야 하며, 한쪽만 바꾸고 나머지를 방치하지 않는다.

## 15. 현재 저장소에 적용된 해석

이 저장소에서는 다음 형태의 작업이 포함될 수 있다.

- 포트폴리오 사이트 문구 수정
- 트레일 러닝 콘셉트 반영
- 게임 섹션 개선
- 모바일 조작성 향상
- GitHub Pages 반영
- 문서 기반 프롬프트 파일 추가

새로운 요청이 들어오면 이 문서의 `Change Item` 구조로 먼저 분해한 뒤 작업을 시작한다.
