# AORR 상태 머신 설계: 개인 프로페셔널 웹사이트 + Games 탭 + 지렁이 게임

이 문서는 `han33u.github.io` 계열의 정적 웹사이트를 GitHub Pages에서 실행 가능한 형태로 만들기 위한 개발 루프를 AORR 상태 머신으로 정리한 설계서다.  
현재 확인 가능한 저장소 상태는 매우 가볍고, 루트에 실제 앱 소스가 아직 없을 수 있으므로, 이 문서는 “설계 우선” 기준으로 작성했다.

현재 확인 가능한 전제:

- 최종 산출물은 HTML, CSS, JavaScript만으로 동작하는 정적 웹사이트다.
- 루트 디렉토리에 최소 `index.html`, `styles.css`, `script.js`가 있어야 한다.
- 게임 코드는 `script.js` 내부에 포함하거나, 별도 JavaScript 파일로 분리할 수 있다.
- GitHub Pages 호환이 최종 기준이다.
- 현재 자료에서 `[게임 추가 기능:]` 항목은 확인되지 않았다. 있으면 [사람 확인 필요]로 반영한다.

## Change Request 管理

- `CHANGE_REQUEST.md` 를 변경 요청의 기준 문서로 사용한다.
- 사용자의 수정 요청은 먼저 `Change Item` 으로 분해한다.
- `AORR.md` 는 루프 상태와 검증 결과를 기록하고, `MEMORY.md` 는 전체 상태를 기억하며, `CHANGE_REQUEST.md` 는 요청 단위를 추적한다.
- 모호한 요청은 추측하지 말고 `[사람 확인 필요]` 로 표시한다.

## Target

프로페셔널 웹사이트 개발 목표:

- 개인 소개, 경력, 프로젝트, 연락처를 담은 정적 프로페셔널 웹사이트를 만든다.
- 모바일과 데스크톱에서 모두 읽기 쉽고 조작 가능한 반응형 레이아웃을 제공한다.
- 상단 내비게이션에 `Games` 탭을 추가하고, 탭 안에서 지렁이 게임을 실행할 수 있게 한다.
- 외부 백엔드 없이 브라우저만으로 완결되게 만든다.

GitHub Pages 배포 목표:

- GitHub Pages가 직접 서빙할 수 있는 정적 파일만 사용한다.
- 절대 경로 의존성을 줄이고 상대 경로 중심으로 구성한다.
- 빌드 단계 없이 배포 가능한 구조를 우선한다.

입력 자료:

- GitHub 저장소의 기존 파일과 디렉토리 구조
- 개인 프로필 콘텐츠: 이름, 소개, 경력, 프로젝트, 링크 [사람 확인 필요]
- 사이트 톤과 디자인 방향 [사람 확인 필요]
- 게임 규칙 및 UX 기준: 이동, 충돌, 점수, 재시작, 터치 조작
- GitHub Pages 배포 설정 정보 [사람 확인 필요]
- 현재 저장소에 있는 추가 요구사항, 특히 `[게임 추가 기능:]` 메모 [사람 확인 필요]

필수 페이지와 섹션:

- 홈/소개 섹션
- 프로필 요약 섹션
- 경력 또는 타임라인 섹션
- 프로젝트 섹션
- 연락처 또는 링크 섹션
- `Games` 탭과 지렁이 게임 화면

Games 탭 및 지렁이 게임 요구사항:

- 상단 내비게이션에서 접근 가능해야 한다.
- 키보드 방향키 또는 `WASD`로 조작 가능해야 한다.
- 모바일에서는 터치 기반 조작이 가능해야 한다.
- 게임 시작, 재시작, 점수 표시가 있어야 한다.
- 충돌 판정, 먹이 생성, 성장 로직이 있어야 한다.
- 화면 크기에 맞게 캔버스 또는 그리드가 반응형으로 동작해야 한다.
- 게임 조작 중 페이지 스크롤과 충돌하지 않도록 해야 한다.
- 현재 자료에 별도 추가 규칙이 있으면 [사람 확인 필요]로 반영한다.

데스크톱 및 모바일 완료 기준:

- 주요 콘텐츠가 데스크톱 폭에서 무너지지 않는다.
- 모바일 폭에서 가로 스크롤이 생기지 않는다.
- 내비게이션이 터치로 조작된다.
- 게임이 모바일에서도 실제로 조작 가능하다.
- 브라우저 콘솔에 에러가 없다.
- GitHub Pages에서 정적 파일만으로 열 수 있다.

## Act

한 번의 개발 루프에서 수행할 최소 작업:

- 실패 원인 하나만 선택한다.
- 그 원인과 직접 연결된 최소 파일만 수정한다.
- 필요한 최소한의 기능만 추가하거나 수정한다.
- 한 루프 안에서 여러 독립 기능을 섞지 않는다.

수정 가능한 파일 범위:

- 현재 루프에서 직접 원인이 된 파일만 수정한다.
- 정적 사이트 기본 구조는 보통 `index.html`, `styles.css`, `script.js`를 우선한다.
- 게임이 별도 파일로 분리되어 있으면 해당 JavaScript 파일만 추가 수정한다.
- GitHub Pages용 보조 파일이 필요할 때만 추가한다. 예: `404.html`, `favicon`, `assets/*`

생성할 수 있는 파일:

- `index.html`
- `styles.css`
- `script.js`
- 필요 시 별도 게임 파일 `game.js` 또는 `snake.js`
- 필요 시 정적 리소스 파일 `assets/*`

실행 가능한 로컬 검증 명령어:

- `python -m http.server`
- `npx serve .`
- `npx prettier --check .` [도구가 있을 때만]
- `npx eslint script.js` [도구가 있을 때만]
- 브라우저 수동 검증: 데스크톱, 모바일 에뮬레이션, 콘솔 확인

## Observe

파일 생성 여부:

- 루트에 필수 파일이 실제로 생성되었는지 확인한다.
- 파일이 링크되었지만 실제로 비어 있거나 경로가 잘못되면 실패로 본다.

HTML, CSS, JavaScript 오류:

- HTML 파싱 오류
- 누락된 `script` 또는 `link`
- CSS 레이아웃 붕괴
- JavaScript 런타임 에러

로컬 웹서버 응답:

- 루트 경로가 200으로 응답하는지 확인한다.
- 정적 자산 경로가 올바른지 확인한다.

브라우저 콘솔 오류:

- 초기 로딩 에러
- 리소스 로드 실패
- 게임 입력 처리 에러
- touch, pointer, keyboard 이벤트 관련 에러

데스크톱 및 모바일 화면:

- 데스크톱에서 레이아웃이 의도대로 배치되는지 본다.
- 모바일에서 텍스트가 잘리는지, 내비게이션이 겹치는지 본다.
- 세로/가로 방향 전환 시 레이아웃이 유지되는지 본다.

키보드 및 터치 게임 조작:

- 키보드 입력이 즉시 반응하는지 확인한다.
- 모바일에서 실제 터치 조작이 먹히는지 확인한다.
- 연속 입력, 빠른 연타, 방향 전환 제한이 정상인지 확인한다.

GitHub Pages 호환성:

- 정적 호스팅에서만 동작하는가
- 빌드 산출물 의존이 없는가
- 루트 경로와 상대 경로가 GitHub Pages에서 깨지지 않는가

## Reason

실패 원인 분류 기준:

- `HTML_STRUCTURE`: DOM 구조, 잘못된 태그 중첩, 링크/섹션 구조 문제
- `CSS_RESPONSIVE`: 반응형 레이아웃, 오버플로, 미디어쿼리, 모바일 표시 문제
- `JAVASCRIPT`: 스크립트 로드 실패, 변수 오류, 초기화 오류, 이벤트 바인딩 오류
- `GAME_LOGIC`: 먹이, 성장, 충돌, 점수 등 게임 규칙 오류
- `GAME_CONTROL`: 키보드/터치 입력, 포커스, 방향 전환, 재시작 조작 오류
- `CONTENT`: 이름, 소개, 경력, 프로젝트 등 개인 콘텐츠 불명확 또는 누락
- `TEST`: 검증 명령 부재, 테스트 실패, 재현 절차 부족
- `ENVIRONMENT`: 로컬 서버, 브라우저, 경로, 파일 시스템, 도구 설치 문제
- `GITHUB_PERMISSION`: 인증, 권한, 저장소 접근, 토큰, 조직 권한 문제
- `DEPLOYMENT`: GitHub Pages 설정, 배포 파일 경로, 공개 노출 문제
- `UNKNOWN`: 원인 식별 불가

판정 규칙:

- 오류가 DOM 구조에 있으면 `HTML_STRUCTURE`
- 반응형만 깨지면 `CSS_RESPONSIVE`
- 자바스크립트 실행 자체가 깨지면 `JAVASCRIPT`
- 게임 규칙이 틀리면 `GAME_LOGIC`
- 입력은 받지만 조작이 실패하면 `GAME_CONTROL`
- 개인 정보가 비어 있거나 검증이 필요하면 `CONTENT`
- 검증 자체가 준비되지 않았으면 `TEST`
- 환경 또는 도구 문제면 `ENVIRONMENT`
- 토큰/권한 문제가면 `GITHUB_PERMISSION`
- Pages 배포 구조가 깨지면 `DEPLOYMENT`
- 어떤 분류에도 명확히 못 넣으면 `UNKNOWN`

## Repeat

반복 원칙:

- 한 번에 하나의 실패 원인만 수정한다.
- 관련된 최소 파일만 변경한다.
- 수정 후 동일한 Verifier를 다시 실행한다.
- 이미 통과한 기능에 대한 회귀 테스트를 함께 실행한다.

권장 반복 순서:

- 구조 실패면 구조만 고친다.
- 스타일 실패면 스타일만 고친다.
- 스크립트 실패면 게임/내비게이션 스크립트만 고친다.
- 게임 실패면 게임 상태, 렌더링, 입력 중 가장 직접적인 부분만 고친다.
- 배포 실패면 경로, 정적 자산, Pages 설정을 고친다.

## Stop

중단 조건:

- 전체 테스트가 통과한 경우
- 최대 Retry에 도달한 경우
- 동일한 오류 fingerprint가 2회 반복된 경우
- 개인정보 또는 콘텐츠 확인이 필요한 경우
- GitHub 인증 또는 배포 권한 문제가 발생한 경우

## Human-in-the-loop

다음 상황에서는 사람이 확인해야 한다:

- 이름, 소개, 경력, 프로젝트 등 개인 콘텐츠가 불명확한 경우
- 기존 콘텐츠 삭제가 필요한 경우
- 외부 분석 도구나 외부 서비스를 추가해야 하는 경우
- GitHub 저장소 설정을 변경해야 하는 경우
- 요구사항이 충돌하는 경우
- `[게임 추가 기능:]` 등 추가 게임 요구가 실제로 존재하는데 현재 자료에서 확인되지 않는 경우

## State Machine

상태 정의:

- `READY`: 다음 루프를 시작할 준비가 된 상태
- `ACTING`: 최소 작업을 수행 중인 상태
- `VERIFYING`: 로컬 검증과 브라우저 확인 중인 상태
- `RETRYING`: 실패 원인을 좁혀 다시 시도하는 상태
- `PASSED`: 해당 루프의 목표가 통과한 상태
- `DEPLOY_READY`: 배포 전 상태로, Pages에 올릴 준비가 끝난 상태
- `DEPLOYING`: GitHub Pages 배포 진행 중인 상태
- `DEPLOYED`: GitHub Pages 배포가 완료된 상태
- `BLOCKED`: 환경, 권한, 의존성 문제로 진행이 막힌 상태
- `HITL_REQUIRED`: 사람이 결정을 내려야 진행 가능한 상태

전이 규칙:

- `READY -> ACTING`: 루프 시작
- `ACTING -> VERIFYING`: 파일 수정 후 검증 시작
- `VERIFYING -> PASSED`: 테스트 통과
- `VERIFYING -> RETRYING`: 테스트 실패
- `RETRYING -> ACTING`: 같은 원인에 대한 최소 수정 수행
- `PASSED -> READY`: 다음 독립 루프로 이동
- `PASSED -> DEPLOY_READY`: 모든 기능 루프가 끝나고 배포 준비 완료
- `DEPLOY_READY -> DEPLOYING`: Pages 배포 시작
- `DEPLOYING -> DEPLOYED`: 배포와 확인이 모두 성공
- `ACTING|VERIFYING|RETRYING -> BLOCKED`: 환경 또는 권한 문제가 해결 불가
- `ACTING|VERIFYING|RETRYING -> HITL_REQUIRED`: 콘텐츠, 정책, 충돌 요구사항이 사람 결정을 필요로 함
- `RETRYING -> BLOCKED`: 같은 실패가 반복되고 원인 격리가 안 됨

## Loop Table

| 단계 | 상태 시작 | 입력 | Act | Observe | 출력 | 테스트 기준 | 다음 상태 |
|---|---|---|---|---|---|---|---|
| 저장소 및 기존 파일 확인 | `READY` | 저장소 URL, 현재 폴더, 기존 파일 목록 | 루트 구조와 기존 파일을 식별하고, 정적 사이트가 들어갈 위치를 확정한다 | 루트에 무엇이 있는지, 앱 소스가 있는지, 민감 파일이 있는지 확인한다 | 작업 범위와 파일 배치 기준 | 루트와 하위 디렉토리 구조가 파악됨 | `PASSED` |
| 정적 사이트 기본 구조 | `READY` | 사이트 목적, 정적 호스팅 제약 | `index.html`, `styles.css`, `script.js`의 최소 골격을 만든다 | HTML이 DOM을 만들고 CSS/JS가 정상 연결되는지 본다 | 렌더 가능한 기본 페이지 | 루트 페이지가 에러 없이 열림 | `PASSED` |

## Detailed Loop Guidance

### 1. 저장소 및 기존 파일 확인

- 입력: 저장소 URL, 현재 파일 목록, 루트 구조
- Act: 현재 자료에서 무엇을 유지하고 무엇을 새로 만드는지 결정한다
- Observe: 기존 콘텐츠, 중복 파일, 민감 파일, empty repo 여부
- 출력: 작업 전제, 초기 디렉토리 전략
- 테스트 기준: 앞으로의 루프가 들어갈 최소 경로가 명확함
- 다음 상태: `PASSED`

### 2. 정적 사이트 기본 구조

- 입력: 최종 파일 요구사항, GitHub Pages 제약
- Act: 최소 HTML/CSS/JS 골격 생성
- Observe: `<link>`와 `<script>` 연결, 콘솔 에러, 200 응답
- 출력: 렌더 가능한 뼈대 페이지
- 테스트 기준: 로컬 서버에서 루트가 뜨고 콘솔 에러가 없음
- 다음 상태: `PASSED`

### 3. 프로페셔널 콘텐츠 영역

- 입력: 개인 소개, 경력, 프로젝트, 링크 [사람 확인 필요]
- Act: Hero, About, Experience, Projects, Contact 섹션 배치
- Observe: 문맥이 자연스러운지, 콘텐츠 누락이 없는지
- 출력: 프로페셔널 정보 구조
- 테스트 기준: 읽기 가능한 정보 계층이 있음
- 다음 상태: `HITL_REQUIRED` 또는 `PASSED`

### 4. 반응형 내비게이션

- 입력: 메뉴 항목, 모바일 우선 레이아웃
- Act: 상단 고정 또는 상단 영역 내비게이션 구현
- Observe: 좁은 화면에서 접힘/오버플로 여부 확인
- 출력: 데스크톱/모바일 공용 내비게이션
- 테스트 기준: 320px 부근에서도 가로 스크롤 없음
- 다음 상태: `PASSED`

### 5. Games 탭

- 입력: 게임 진입 요구, 탭 네이밍
- Act: `Games` 탭을 추가하고 게임 화면으로 연결
- Observe: 탭 전환, 포커스, URL 유지 여부
- 출력: 게임 진입점
- 테스트 기준: 탭 클릭 또는 키보드 접근이 가능함
- 다음 상태: `PASSED`

### 6. 지렁이 게임 핵심 로직

- 입력: 그리드 크기, 속도, 점수, 충돌 규칙 [사람 확인 필요]
- Act: 게임 상태 머신, 이동, 먹이, 성장, 충돌 로직 구현
- Observe: 같은 입력에 같은 결과가 나오는지, 예외가 없는지
- 출력: 동작하는 게임 엔진
- 테스트 기준: 먹이를 먹으면 길이가 늘고, 벽/자기몸 충돌 시 종료됨
- 다음 상태: `PASSED`

### 7. 키보드 조작

- 입력: 방향키, `WASD`, 재시작 키
- Act: 입력을 게임 상태에 매핑
- Observe: 반대 방향 금지, 입력 누락, 포커스 손실 여부
- 출력: 키보드 조작 가능한 게임
- 테스트 기준: 키보드로 안정적으로 플레이 가능
- 다음 상태: `PASSED`

### 8. 모바일 터치 조작

- 입력: 터치/스와이프/버튼 방식 [사람 확인 필요]
- Act: 모바일 입력을 게임 컨트롤에 연결
- Observe: 터치가 스크롤과 충돌하는지, 화면 회전 후 유지되는지
- 출력: 모바일 조작 인터페이스
- 테스트 기준: 실제 모바일 또는 에뮬레이터에서 플레이 가능
- 다음 상태: `PASSED`

### 9. 게임 UI 및 점수

- 입력: 점수 규칙, 시작/재시작 UX
- Act: HUD, 상태 메시지, 점수, 재시작 버튼 구현
- Observe: 점수 동기화, 시작 전/게임 오버 표기
- 출력: 읽기 쉬운 게임 인터페이스
- 테스트 기준: 점수와 상태가 항상 일치함
- 다음 상태: `PASSED`

### 10. 접근성과 반응형 검증

- 입력: 폰트, 대비, 포커스, 다양한 화면 크기
- Act: 접근성 및 반응형 버그를 수정
- Observe: 키보드 포커스, 대비, 오버플로, 작은 화면 가독성
- 출력: 다기기 호환 레이아웃
- 테스트 기준: 데스크톱과 모바일 모두에서 읽고 조작 가능
- 다음 상태: `PASSED`

### 11. GitHub Pages 호환성 검증

- 입력: 정적 호스팅 규칙, 상대 경로
- Act: 절대 경로 제거, 자산 경로 점검, 정적 구성 확인
- Observe: 로컬과 Pages 경로 차이, 404 가능성
- 출력: 배포 준비 상태
- 테스트 기준: 별도 서버 없이 Pages에서 실행 가능
- 다음 상태: `DEPLOY_READY`

### 12. 배포

- 입력: GitHub 저장소 권한, Pages 설정
- Act: Pages 배포 수행
- Observe: 배포 URL, 자산 로드, 404, 캐시 문제
- 출력: 공개 배포 사이트
- 테스트 기준: 프로덕션 URL에서 정상 동작
- 다음 상태: `DEPLOYED`

## Recommended First Loop

가장 안전한 첫 번째 루프는 다음이다.

- 저장소 및 기존 파일 확인
- 정적 사이트 기본 구조
- 파일 생성만으로 끝나는 최소 뼈대 검증

이유:

- 개인 콘텐츠나 게임 규칙을 아직 확정하지 않아도 진행할 수 있다.
- GitHub 권한이나 배포 권한이 없어도 로컬에서 검증 가능하다.
- 이후의 모든 루프가 의존하는 파일 골격을 먼저 안정화할 수 있다.

첫 루프의 성공 정의:

- 루트에 `index.html`, `styles.css`, `script.js`가 존재한다.
- 로컬 서버에서 빈 에러 없이 페이지가 열린다.
- 아직 확정되지 않은 개인 정보와 게임 규칙은 [사람 확인 필요]로 남긴다.

## Notes

- 현재 자료에서 `[게임 추가 기능:]`이 보이지 않아, 해당 항목은 별도 확인 전까지 게임 루프에 직접 반영하지 않는다.
- GitHub 인증이 필요할 때는 토큰을 출력하지 않고, 파일에 영구 저장하지 않으며, 커밋 대상에도 넣지 않는다.
- 설계 단계에서는 코드 수정, 테스트 실행, 배포를 하지 않는다.
- 최신 Step 8 변경 요청은 현재 `CHANGE_REQUEST.md`/`MEMORY.md`/배포 상태와 일치하는 워크플로 검증 요청으로 확인되었고, 별도 Change Item 추가 없이 기록만 갱신했다.
- 최근 포트폴리오 디자인 요청은 CSS 중심의 시각적 개선으로 처리했고, 구조와 게임 로직은 그대로 보존했다.
- Swiss UTMB 배경 요청은 생성 이미지 자산을 추가하고 히어로 배경과 색감을 알프스 톤으로 바꾸는 디자인 Change Item으로 처리했다.
- 최근 전체 페이지 재디자인 요청은 같은 Swiss UTMB 컨셉을 본문, 카드, 내비게이션, 게임, 푸터까지 확장하는 CSS 중심 Change Item으로 처리했다.
- 최근 사용자 요청으로 Games 섹션에 등산 게임이 추가되었고, 현재는 지렁이 게임과 등산 게임을 함께 제공하는 방향으로 확장되었다.

## Self-Correcting TDD Loop

이 섹션은 GitHub Pages에서 동작하는 정적 HTML, CSS, JavaScript 웹사이트를 위해 Verifier 중심으로 실패를 좁히고, 한 번에 하나씩 수정하는 Self-Correcting TDD 루프를 정의한다.

### 현재 환경에서 확인된 검증 도구

현재 작업 공간에서 `Get-Command`로 확인한 결과:

- 확인됨: `python.exe`, `python3.exe`
- 확인되지 않음: `node`, `npm`, `npx`, `git`, `claude`

즉, 현재 상태에서는 Python 기반 로컬 정적 서버와 파일 검사 위주의 검증만 즉시 설계할 수 있다. npm 기반 검증은 실제 도구가 확인된 뒤에만 사용한다.

현재 저장소 구조에서 확인된 파일:

- `AORR.md`
- `samsung-agent-education/README.md`
- `samsung-agent-education/github_token.txt`
- `samsung-agent-education/claude-code-cli/claude_auto_install.md`

### Verifier 우선순위

1. 파일 존재 및 경로 검증
2. HTML 구조 검증
3. CSS 반응형 검증
4. JavaScript 정적/런타임 검증
5. 지렁이 게임 동작 검증
6. 로컬 정적 서버 응답 검증
7. 브라우저 viewport 검증
8. GitHub Pages 호환성 검증
9. Claude Code CLI 기반 보조 Verifier 검증

### Claude Code CLI 사용 원칙

- Claude Code CLI가 실제로 PATH에서 발견되면 독립 Verifier로 사용한다.
- 먼저 `Sonnet 5` 사용 가능 여부를 확인한다.
- `Sonnet 5`가 실제로 사용 가능할 때만 `Sonnet 5`를 사용한다.
- `Sonnet 5`가 불가능하면, 현재 사용 가능한 Sonnet 모델의 실제 모델명을 기록하고 그 모델을 사용한다.
- 모델명은 추정하지 않고, CLI 출력 또는 공식 응답에서 확인된 값만 기록한다.
- 현재 환경에서는 `claude`가 확인되지 않았으므로, 이 단계는 실행 전제 설계만 둔다.

### Self-Correcting 루프 상태

- `PRECHECK`
- `VERIFYING`
- `FAIL_CLASSIFIED`
- `RETRYING`
- `PASSED`
- `BLOCKED`
- `HITL_REQUIRED`

### 루프 입력

- 현재 저장소 구조
- `AORR.md`의 대상/실행 기준
- 실제 존재하는 검증 도구 목록
- 브라우저 콘솔 출력
- 파일 경로와 라인 정보
- GitHub Pages 호환 제약
- 지렁이 게임 요구사항

### 루프 출력

- 실패 원인 분류
- 오류 fingerprint
- 수정 대상 파일 목록
- 재실행할 동일 Verifier
- 사람 확인 필요 여부
- 다음 상태

### 실패 로그 스키마

실패가 발생하면 아래 항목을 반드시 모은다.

- 실행 명령어
- exit code
- 실패한 검증 항목
- 핵심 오류 메시지
- 관련 파일과 라인
- 브라우저 콘솔 메시지
- 오류 fingerprint
- 분류된 실패 원인
- 재시도 가능 여부

### 실패 원인 분류

아래 분류만 사용한다.

- `HTML_STRUCTURE`
- `CSS_RESPONSIVE`
- `JAVASCRIPT`
- `GAME_LOGIC`
- `GAME_CONTROL`
- `CONTENT`
- `TEST`
- `ENVIRONMENT`
- `GITHUB_PERMISSION`
- `DEPLOYMENT`
- `UNKNOWN`

### Verifier별 체크리스트

#### 1) 기본 파일 검증

목표:

- 루트 `index.html` 존재
- CSS 및 JavaScript 연결
- 잘못된 로컬 파일 경로 없음
- 대소문자 불일치 없음
- 절대 로컬 경로 미사용

권장 관찰:

- 파일 존재 여부
- 상대 경로만 사용했는지
- GitHub Pages에서 깨질 로컬 경로가 없는지

실패 분류:

- 파일 누락 또는 잘못된 링크는 `HTML_STRUCTURE`
- 로컬 절대 경로는 `DEPLOYMENT` 또는 `HTML_STRUCTURE`
- 대소문자 문제는 `DEPLOYMENT`

#### 2) HTML 검증

목표:

- 기본 문서 구조
- `title`, `meta viewport`
- 시맨틱 태그
- 내비게이션 링크
- `Games` 영역
- 이미지 `alt`
- 깨진 내부 링크

실패 분류:

- 구조 누락은 `HTML_STRUCTURE`
- 콘텐츠가 비어 있으면 `CONTENT`

#### 3) CSS 검증

목표:

- 데스크톱 화면
- 태블릿 화면
- 모바일 화면
- 가로 스크롤 여부
- 내비게이션 및 `Games` UI 반응형 동작

실패 분류:

- 레이아웃 붕괴, overflow, 브레이크포인트 문제는 `CSS_RESPONSIVE`

#### 4) JavaScript 검증

- 권한/토큰 문제는 `GITHUB_PERMISSION`

### Retry 정책

- 한 오류에 대해 최대 3회 시도
- 동일 fingerprint가 2회 반복되면 중지
- Retry마다 다음을 남긴다:
  - 가설
  - 변경 파일
  - 실행 명령어
  - 결과 기록
- 환경 또는 권한 문제는 코드 수정으로 해결하지 않는다
- 이미 통과한 기능은 회귀로 다시 확인한다

### 최소 수정 원칙

- 한 Retry에서는 하나의 원인만 수정
- 관련 파일만 수정
- 테스트 삭제 금지
- 검증 기준 완화 금지
- 전체 사이트 재작성 금지
- 외부 프레임워크로 임의 전환 금지

### 중단 조건

- 전체 테스트가 통과한 경우
- 최대 Retry에 도달한 경우
- 동일 오류 fingerprint가 2회 반복된 경우
- 개인정보나 콘텐츠 확인이 필요한 경우
- GitHub 인증 또는 배포 권한 문제가 발생한 경우

### 권장 실행 순서

1. 현재 파일 구조와 존재 도구 확인
2. 기본 파일 검증
3. HTML 검증
4. CSS 검증
5. JavaScript 검증
6. 지렁이 게임 검증
7. 로컬 서버 검증
8. 브라우저 viewport 검증
9. GitHub Pages 호환성 검증
10. 필요 시 Claude Code CLI를 독립 Verifier로 사용

### Verifier 중심 판정 기준

각 루프는 다음 세 가지 질문으로 종료한다.

1. 실패가 실제로 재현되는가
2. 실패 원인이 한 분류로 좁혀졌는가
3. 수정 후 같은 Verifier가 다시 통과하는가

이 세 가지가 모두 충족되면 `PASSED`로 넘긴다.
하나라도 불명확하면 `RETRYING` 또는 `HITL_REQUIRED`로 둔다.
